# EHUB-604 — Provider Strategy & Anti-Corruption

**Status:** DRAFT — awaiting Architect review.

## Principle

> The domain speaks **one internal language**. Each payment provider is an external dialect translated at the edge by an **adapter**. Provider payloads never leak into the Payment aggregate (L8).

## Layers

```text
        Domain (Payment aggregate)
                 ▲   normalized types only
                 │
   Application (orchestration, uses IPaymentProvider)
                 ▲   ProviderPaymentResult / ProviderRefundResult
                 │
   Anti-Corruption Adapter (per provider)  ── maps provider ⇄ internal
                 ▲   raw HTTP/JSON/signature
                 │
        Provider SDK / REST (Stripe, Payriff, ...)
```

## Provider abstraction (names only — no code this sprint)

```text
interface IPaymentProvider
  ProviderKey                         // Stripe | Payriff | Manual | Test
  CreatePayment(request)  -> ProviderPaymentResult
  Refund(request)         -> ProviderRefundResult
  VerifySignature(headers, rawBody) -> bool          (L6)
  ParseWebhook(rawBody)   -> ProviderPaymentResult   (L8)
```

## Normalized internal types (the only thing the domain sees)

```text
ProviderPaymentResult
  ProviderKey
  ProviderPaymentId
  Outcome: Pending | Succeeded | Failed | Cancelled | Expired
  Amount / CurrencyId          (as reported — validated vs Payment.Amount)
  FailureReason?               (normalized code + message, NOT raw payload)
  EventId                      (provider event id — used for dedupe, L2)
  OccurredAtUtc
  RawPayloadRef?               (opaque, stored audit-only)

ProviderRefundResult
  ProviderRefundId
  Outcome: Succeeded | Failed | Pending
  Amount / CurrencyId
  FailureReason?
```

## Provider selection

| Aspect | Rule |
|--------|------|
| Selection | `Provider` chosen at Payment creation (config/company policy). Stored on Payment. |
| Routing | Webhook endpoint is provider-scoped: `/payments/webhook/{provider}` → correct adapter. |
| Config | Secrets/keys per provider in configuration, never in domain. |
| Fallback | `Manual`/`Test` providers for local/dev + admin-recorded payments. |

## Anti-corruption rules (locked candidates)

| # | Rule |
|---|------|
| ACL-1 | Adapter is the **only** place that references a provider SDK/type. |
| ACL-2 | Adapter output is a **normalized** result; no provider enum/string leaks past it (L8). |
| ACL-3 | Raw payloads may be stored **audit-only** (opaque JSON), never mapped to domain fields. |
| ACL-4 | Signature verification lives in the adapter (`VerifySignature`) (L6). |
| ACL-5 | Amount/currency from provider is **validated against** `Payment.Amount` — never overwrites it (L1, BR-PAY-001). |
| ACL-6 | Idempotency key is generated by the application and passed **into** the adapter, not by the provider. |
| ACL-7 | Provider errors are mapped to a normalized `FailureReason` catalog; unknowns → `provider_error_unknown`. |

## Why anti-corruption here

- Adding a second provider (e.g. Payriff after Stripe) must **not** touch the Payment aggregate or Booking.
- Provider breaking-changes are contained in one adapter.
- Tests can substitute a fake `IPaymentProvider` without any HTTP.
